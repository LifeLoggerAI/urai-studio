## Discovery findings
- Framework: Next.js app exists in apps/studio (App Router) with static export enabled.
- Runtime/package manager: Node + pnpm workspace.
- Build system: next build under apps/studio; monorepo scripts fan out.
- Hosting target/site: .firebaserc maps project urai-studio with hosting target default->urai-studio.
- Data/auth layers: Firebase client/admin modules exist in multiple locations; duplicated legacy paths detected.
- Tests: Playwright smoke test exists (apps/studio/tests/studio.smoke.spec.ts).
- Deployment scripts: root firebase config + multiple shell deploy helpers in scripts/ and tmp/.
- Duplicate/legacy folders: .bak/, many *.bak files, parallel app trees in apps/studio/app and apps/studio/src/app.
- Risks:
  1) Conflicting source-of-truth between app/ and src/app.
  2) Static export currently disables API routes; requested /api/system/* cannot work with export-only config.
  3) tsconfig path alias points to ./src/* while current active app seems rooted at ./app.
  4) firebase-admin currently requires service account env vars; needs ADC-safe fallback.
  5) No canonical module registry and no required system endpoints.
- Proposed implementation plan:
  1) Establish canonical lib/studio contract layer and integration bridge.
  2) Build a shared app shell + navigation + module pages/routes preserving existing routes.
  3) Add API system endpoints and openapi/capability docs.
  4) Harden Firebase/env config + .env.example.
  5) Add tests (unit + API schema-level) and smoke script.
  6) Run typecheck/lint/test/build and produce final proof report.
